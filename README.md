# ai-reviewer

> **Production-grade AI-to-AI code review engine.**
> Two heterogeneous LLMs cross-validate every diff in parallel. Both must agree with high confidence — or the pipeline is blocked.

Designed for **pure-AI coding environments** where all code is generated by agents and there are no human reviewers in the loop. The tool acts as a hard CI gate: no mercy, no override, just exit codes.

---

## Why This Exists

When AI agents write all the code, a single model reviewing its own output creates a blind spot — it can hallucinate, self-justify, or miss entire vulnerability classes. This tool forces **two architecturally different models** (MiniMax + DeepSeek) to independently review the same diff. They must both:

1. Produce valid structured JSON (with automatic self-correction on failure)
2. Report `"verdict": "pass"`
3. Report `confidence >= 0.90`

Only when all three conditions are met simultaneously does the gate open (exit code `0`).

---

## Features

| Feature | Implementation |
|---|---|
| **AST-level context** | `tree-sitter` parses full source files, extracts changed symbols and call graphs — not raw text chunks |
| **Heterogeneous cross-validation** | MiniMax + DeepSeek run concurrently via `tokio::join!`, each unaware of the other's output |
| **Self-correction retry loop** | On JSON parse failure, the exact `serde` error is sent back to the model; up to 3 correction attempts per model |
| **Strict typed output** | All LLM output is deserialized into Rust structs — no `serde_json::Value` fallbacks, no silent data loss |
| **Prompt caching** | The security policy (can be 50–100 KB) is sent as a stable system prompt prefix, cached server-side between CI runs |
| **Exit code as CI gate** | `0` = both pass, `1` = blocked, `2` = fatal error. Works with any CI system with zero integration overhead |

```bash
cargo install --path .
```

## Usage

### Basic

```bash
ai-reviewer \
  --diff <path-to.diff> \
  --policy <security-policy.md> \
  --source-root <repo-root>
```

### Read diff from stdin (typical CI usage)

```bash
git diff HEAD~1 | ai-reviewer \
  --diff - \
  --policy ./policy/security.md \
  --source-root .
```

### All options

```
USAGE:
  ai-reviewer [OPTIONS] --diff <PATH> --policy <PATH>

OPTIONS:
  -d, --diff <PATH>              Unified diff file, or "-" for stdin
  -s, --source-root <PATH>       Repo root for full-file AST resolution [default: .]
  -p, --policy <PATH>            Security/coding policy Markdown file [required]
  -t, --threshold <FLOAT>        Confidence gate threshold, 0.0–1.0 [default: 0.90]
  -o, --output <PATH>            Report output path [default: ./review-report.md]
      --max-retries <N>          Self-correction retries per model [default: 3]
      --model-minimax <ID>       MiniMax model ID [default: MiniMax-M2.5]
      --model-deepseek <ID>      DeepSeek model ID [default: deepseek-chat]
      --minimax-base-url <URL>   Override MiniMax API endpoint
      --deepseek-base-url <URL>  Override DeepSeek API endpoint
  -v, --verbose                  Enable debug tracing (or set RUST_LOG=debug)
  -h, --help                     Print help
  -V, --version                  Print version

ENVIRONMENT:
  MINIMAX_API_KEY      Required. MiniMax API key.
  DEEPSEEK_API_KEY     Required. DeepSeek API key.
  MINIMAX_MODEL        Override --model-minimax
  DEEPSEEK_MODEL       Override --model-deepseek
  MINIMAX_BASE_URL     Override --minimax-base-url
  DEEPSEEK_BASE_URL    Override --deepseek-base-url
  RUST_LOG             Tracing filter, e.g. "ai_reviewer=debug"
```

## The Policy File

The `--policy` argument points to a Markdown file that becomes the LLM system prompt. It defines what the reviewers look for. This file is cached server-side — send it once, reuse across CI runs.

**Example `policy/security.md`:**

```markdown
## Security Rules

- SEC-001: Never log credentials, tokens, or PII
- SEC-002: All SQL queries must use parameterized statements
- SEC-003: File paths must be validated against an allowlist before use
- SEC-004: External HTTP calls must go through the approved proxy client

## Code Quality Rules

- LOGIC-001: Every error must be handled or explicitly propagated — no silent swallows
- LOGIC-002: All public functions must validate their inputs at the boundary
- LOGIC-003: Async tasks must not hold locks across await points
```

The LLM must reference these rule IDs in its findings (e.g. `"rule_id": "SEC-001"`). This makes the output auditable and traceable.

## Output

### Stdout (always printed)

```
[ai-reviewer] Verdict: PASS  |  MiniMax: PASS (94.0%)  |  DeepSeek: PASS (91.0%)  |  Findings: 2
[ai-reviewer] Gate PASSED — full report: ./review-report.md
```

### Report file (always written, default: `review-report.md`)

A Markdown cross-comparison report including:
- Overall verdict and gate failure analysis (if blocked)
- Confidence summary table per model
- Merged + deduplicated findings table (sorted by severity: CRITICAL → INFO)
- Per-finding detail cards with file locations and suggestions
- Per-model reasoning sections

### Exit codes

| Code | Meaning |
|---|---|
| `0` | Gate PASSED — both models approved with confidence ≥ threshold |
| `1` | Gate BLOCKED — conflict, low confidence, parse failure, or defects found |
| `2` | Fatal error — bad input paths, missing API keys, unrecoverable failure |

## CI/CD Integration

### GitHub Actions

```yaml
- name: AI Code Review
  env:
    MINIMAX_API_KEY: ${{ secrets.MINIMAX_API_KEY }}
    DEEPSEEK_API_KEY: ${{ secrets.DEEPSEEK_API_KEY }}
  run: |
    git diff origin/main...HEAD > /tmp/pr.diff
    ./ai-reviewer \
      --diff /tmp/pr.diff \
      --policy ./policy/security.md \
      --source-root . \
      --output /tmp/review-report.md

- name: Upload Review Report
  if: always()
  uses: actions/upload-artifact@v4
  with:
    name: ai-review-report
    path: /tmp/review-report.md
```

### GitLab CI

```yaml
ai-review:
  script:
    - git diff origin/main...HEAD > pr.diff
    - ./ai-reviewer --diff pr.diff --policy policy/security.md --source-root .
  artifacts:
    when: always
    paths:
      - review-report.md
```

## How It Works (Pipeline Overview)

```
git diff
    │
    ▼
1. DIFF PARSER        Parse unified diff → per-file hunks + line ranges
    │
    ▼
2. AST EXTRACTOR      tree-sitter parses full source files
                      → maps changed lines to enclosing functions/structs
                      → builds lightweight call graph (caller → callees)
    │
    ▼
3. PROMPT ASSEMBLER   System block (policy, cached) + User block (AST context + diff)
    │
    ├─────────────────────────────────┐
    ▼                                 ▼
4. MINIMAX REVIEWER           DEEPSEEK REVIEWER
   (concurrent via             (concurrent via
    tokio::join!)               tokio::join!)
   Self-correction loop        Self-correction loop
    │                                 │
    └─────────────┬───────────────────┘
                  ▼
5. CONSENSUS ENGINE   Both confident + both pass → gate open
                      Any failure → gate blocked + report
                  │
                  ▼
6. REPORTER           Markdown report written to disk
                      Exit code 0 or 1
```

## For AI Agents Using This Tool

> This section is for AI coding agents (Claude Code, Cursor, Copilot, custom agents) that need to invoke or integrate `ai-reviewer` programmatically.

### Minimal invocation pattern

```bash
# 1. Generate the diff from your working changes
git diff HEAD > /tmp/changes.diff

# 2. Run the reviewer (blocks until both models respond)
ai-reviewer \
  --diff /tmp/changes.diff \
  --policy /path/to/policy.md \
  --source-root /path/to/repo \
  --output /tmp/review.md

# 3. Check exit code — this is your gate signal
# exit 0 → safe to proceed (commit, deploy, etc.)
# exit 1 → blocked; parse /tmp/review.md for structured findings
# exit 2 → configuration error; do not retry without fixing the error
```

### Reading the report programmatically

The report at `--output` is structured Markdown. To extract findings, look for the table under `## Findings (merged & deduplicated)` with columns: `# | Severity | Rule | Location | Description`.

Each finding has a `rule_id` (e.g. `SEC-001`) and a `Location` in the form `` `file.rs:42` ``. Parse these to know exactly which lines to fix.

### Self-healing loop pattern (for autonomous agents)

```
loop:
  1. Generate code changes
  2. Run ai-reviewer → capture exit code + report path
  3. If exit 0 → done ✓
  4. If exit 1 → read findings from report
               → for each CRITICAL/HIGH finding:
                   fix the code at the indicated location
               → go to step 1
  5. If exit 2 → check API keys and file paths, do not loop
  6. Abort after N iterations to avoid infinite loops
```

### What the models receive

Each model receives:
- Your full security policy (system prompt, cached)
- The unified diff of changed files
- The **full source definition** of every changed function/class (not just the diff lines)
- The **call graph**: which functions the changed code calls, and their full definitions

This means the models have complete semantic context, not just text fragments.

### Tuning for your use case

| Scenario | Recommended flag |
|---|---|
| Strict security gate (default) | `--threshold 0.90` |
| Experimental / draft code | `--threshold 0.75` |
| Maximum strictness | `--threshold 0.95` |
| Debugging model outputs | `--verbose` + `RUST_LOG=debug` |
| Fast iteration (fewer retries) | `--max-retries 1` |
